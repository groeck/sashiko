# DESIGN: Kiro CLI Provider Support

## Context

Sashiko supports API-backed and CLI-backed providers through the generic
`AiProvider` trait in `src/ai/mod.rs`. The existing `claude-cli` and
`codex-cli` providers are the closest local patterns:

- `src/ai/claude_cli.rs` converts an `AiRequest` into a text prompt, shells out
  to a local CLI, and parses either final content or a synthetic JSON tool-call
  envelope.
- `src/ai/codex_cli.rs` reuses the Claude CLI prompt and response parser, then
  adapts Codex JSONL events into plain text.
- `src/reviewer.rs` maps API/CLI providers used by the parent process to a
  `stdio-*` provider in the child review binary, so the child asks the parent
  for model completions instead of calling external services directly.

Kiro CLI exposes an Agent Client Protocol (ACP) mode:

```bash
kiro-cli acp
```

ACP is JSON-RPC 2.0 over the subprocess stdin/stdout streams. The `kiro-cli
acp` subcommand accepts `--agent`, `--model`, `--trust-all-tools`, and
`--verbose`. ACP is a better fit than `kiro-cli chat --no-interactive`
because stdout is a machine protocol instead of terminal UI output, and
because `chat --no-interactive` can still attempt Kiro-native tools and
require approvals that cannot be granted in non-interactive mode.

## Design Decisions

| Decision | Choice |
|---|---|
| Provider name | `"kiro-cli"` |
| Client architecture | New `KiroCliProvider` in `src/ai/kiro_cli.rs`, matching the existing CLI-backed provider pattern |
| Kiro mode | `kiro-cli acp` |
| Prompt transport | Full Sashiko prompt sent as ACP `session/prompt` text content |
| Response parsing | Reuse `claude_cli::parse_inner_response()` for final content and Sashiko JSON `tool_calls` |
| Native Kiro tools | Disabled by default through an isolated generated Kiro agent |
| Safety backstop | Generated deny-all `preToolUse` hook in the temporary Kiro workspace |
| Custom agent override | Optional `[ai.kiro_cli].agent`; treated as an advanced escape hatch |
| Token usage | Synthesized with `TokenBudget` because Kiro ACP does not expose token counts |
| Stdio child mapping | Parent provider maps to `stdio-claude` for child review workers |
| Session reuse | Not supported; every `generate_content()` call is stateless |

## Provider Compatibility

References:

- Kiro CLI command reference: https://kiro.dev/docs/cli/reference/cli-commands/
- Kiro custom agents: https://kiro.dev/docs/cli/custom-agents/configuration-reference/
- Kiro authentication: https://kiro.dev/docs/cli/authentication/

The provider implements the ACP flow directly: spawn `kiro-cli acp`, perform
`initialize`, create a fresh session, send one `session/prompt`, collect
`session/update` text chunks, and stop the subprocess. Because Sashiko already
sends full conversation history on every `generate_content()` call, ACP
sessions are used as one-shot completion requests and are not reused across
turns.

Tests use a small fake ACP subprocess so normal CI does not require a Kiro
account, network access, or `KIRO_API_KEY`.

## Goals

1. Add a provider named `kiro-cli`.
2. Use local `kiro-cli acp` as a stateless completion backend.
3. Reuse the existing Sashiko text prompt and synthetic tool-call protocol.
4. Keep Kiro's native tools disabled by default. Sashiko's own `ToolBox`
   remains the only tool execution layer used by the review worker.
5. Preserve Sashiko token/cost accounting as well as possible even though
   Kiro ACP output does not expose provider token usage.
6. Keep configuration minimal and compatible with environment overrides.

## Non-goals

- Do not expose Kiro native shell/write tools as part of the first
  implementation.
- Do not add an OpenAI-compatible Kiro API provider.
- Do not implement session resume. Each `generate_content` call is
  independent because Sashiko already sends full conversation history in
  the `AiRequest`.
- Do not parse Kiro internal logs or tool-progress output as first-class
  model events.

## Files

### `src/settings.rs`

Add provider-specific settings:

```rust
#[derive(Debug, Deserialize, Clone)]
#[allow(unused)]
pub struct KiroCliSettings {
    #[serde(default = "default_kiro_cli_binary")]
    pub binary: String,
    #[serde(default)]
    pub agent: Option<String>,
    #[serde(default = "default_kiro_cli_context_window")]
    pub context_window_size: usize,
}
```

Add `pub kiro_cli: Option<KiroCliSettings>` to `AiSettings`.

Suggested defaults:

- `binary = "kiro-cli"`
- `agent = None`
- `context_window_size = 200_000`

`api_timeout_secs` continues to come from the existing global
`[ai] api_timeout_secs` setting.

Advanced users may set `agent`, but the default path uses an isolated
temporary no-tool agent generated by the provider itself.

### `src/ai/kiro_cli.rs`

Create `src/ai/kiro_cli.rs` with:

```rust
pub struct KiroCliProvider {
    pub model: String,
    pub binary: String,
    pub agent: Option<String>,
    pub context_window_size: usize,
    pub timeout_secs: u64,
}
```

Implement `AiProvider`:

1. Build the full Sashiko prompt with `claude_cli::build_prompt(&request)`.
2. If `agent` is unset, create a temporary directory with:

   ```text
   <tmp>/.kiro/agents/sashiko-provider.json
   ```

   The generated agent should look like:

   ```json
   {
     "name": "sashiko-provider",
     "description": "Stateless Sashiko completion backend. Native Kiro tools are disabled.",
     "prompt": "Follow the user-provided instructions exactly.",
     "mcpServers": {},
     "tools": [],
     "allowedTools": [],
     "resources": [],
     "includeMcpJson": false
   }
   ```

   The agent prompt is intentionally minimal so it does not override the
   Sashiko review protocol already embedded in the `build_prompt` output.

   Also create a deny-all pre-tool hook in the temporary workspace:

   ```text
   <tmp>/.kiro/hooks/deny-all-tools.sh
   ```

   ```bash
   #!/bin/sh
   echo "Kiro native tools are disabled for the Sashiko provider" >&2
   exit 1
   ```

   and wire it into the generated agent:

   ```json
   "hooks": {
     "preToolUse": [
       {
         "command": ".kiro/hooks/deny-all-tools.sh"
       }
     ]
   }
   ```

   The hook is a defensive backstop. With `tools: []`, Kiro should not have
   native tools available, but the hook makes accidental future config drift
   fail closed.

   Run Kiro with `current_dir` set to the temporary directory and `--agent
   sashiko-provider`. Kiro docs say local agents are discovered from
   `.kiro/agents` under the current working directory and take precedence
   over global agents.
3. If `agent` is set in config, use that configured agent and the process
   current directory. This is an explicit advanced escape hatch and should
   be documented as less isolated.
4. Spawn:

   ```text
   kiro-cli acp --agent <agent> --model <model>
   ```

5. Speak JSON-RPC 2.0 over the subprocess stdin/stdout:
   - `initialize` with protocol version 1 and empty client capabilities.
   - `session/new` with `cwd = "."` and `mcpServers = []`.
   - `session/prompt` with the full `build_prompt` output as text content.
6. Collect `session/update` notifications until the `session/prompt`
   response arrives. Concatenate `agent_message_chunk` /
   `AgentMessageChunk` text content into the completion text.
7. Apply a timeout using `tokio::time::timeout`.
8. On JSON-RPC error, timeout, missing session ID, or subprocess failure,
   fail with a concise sanitized error.
9. Parse the collected completion text with
   `claude_cli::parse_inner_response`.
10. Since Kiro does not expose ACP token usage, synthesize approximate
    usage:

    ```rust
    AiUsage {
        prompt_tokens: TokenBudget::estimate_tokens(&prompt),
        completion_tokens: TokenBudget::estimate_tokens(&completion_text),
        total_tokens: prompt_tokens + completion_tokens,
        cached_tokens: None,
    }
    ```

The provider's `estimate_tokens` method also uses `TokenBudget` over the
request content, system prompt, tool definitions, and previous tool calls.

### `src/ai/mod.rs`

- Add `pub mod kiro_cli;`
- Add a `create_provider` arm:

```rust
"kiro-cli" => {
    let cfg = settings.ai.kiro_cli.as_ref();
    Ok(Arc::new(kiro_cli::KiroCliProvider {
        model: settings.ai.model.clone(),
        binary: cfg.map(|c| c.binary.clone()).unwrap_or_else(|| "kiro-cli".to_string()),
        agent: cfg.and_then(|c| c.agent.clone()),
        context_window_size: cfg.map(|c| c.context_window_size).unwrap_or(200_000),
        timeout_secs: settings.ai.api_timeout_secs,
    }))
}
```

### `src/reviewer.rs`

Extend the child-provider mapping so Kiro reviews flow through the parent
process:

```rust
"claude" | "stdio-claude" | "claude-cli" | "codex-cli" | "kiro-cli" => "stdio-claude",
```

This keeps Kiro calls in the parent process and lets the child review
binary use the existing stdio JSON protocol.

## Tests

Add focused unit tests. Do not run real `kiro-cli` in normal tests.

| # | Test | Verifies |
|---|---|---|
| 1 | Factory | `create_provider` returns a `kiro-cli` provider and reports the configured model and context window. |
| 2 | Command args (default) | Spawn args include `acp`, `--agent sashiko-provider`, and `--model <model>` in that order. |
| 3 | Command args (custom agent) | When `[ai.kiro_cli].agent` is set, `--agent <that>` is used instead. |
| 4 | Isolated workspace agent JSON | Generated agent has `"tools": []`, `"allowedTools": []`, `"mcpServers": {}`, `"includeMcpJson": false`, a minimal `prompt`, and a `preToolUse` hook pointing at `.kiro/hooks/deny-all-tools.sh`. |
| 5 | Deny-all hook | The generated hook script exits non-zero and writes the expected message to stderr. |
| 6 | ACP chunk extraction (snake_case) | `session/update` notifications with `sessionUpdate: "agent_message_chunk"` yield the embedded text. |
| 7 | ACP chunk extraction (CamelCase) | `session/update` notifications with `sessionUpdate: "AgentMessageChunk"` yield the embedded text, including array-of-parts content. |
| 8 | ACP ignores non-text updates | Notifications with other `sessionUpdate` values (e.g., `tool_call`) return no text. |
| 9 | Tool-call parsing | Kiro output containing `{"tool_calls":[{"id":"c1","function_name":"read_file","arguments":{...}}]}` parses into `AiResponse.tool_calls` via the shared parser. |
| 10 | Plain content parsing | `{"content":"..."}` parses into `AiResponse.content`. |
| 11 | Raw text fallback | Non-JSON output is returned as raw content through the shared parser. |
| 12 | Token estimation | `estimate_tokens` returns a positive count for a non-empty request. |
| 13 | End-to-end fake ACP | A fake ACP subprocess handles `initialize`, `session/new`, and `session/prompt`, emits a `session/update` text chunk, and Sashiko parses the expected `content`. |

## Error Handling

Handle these cases explicitly:

- Missing binary: `Failed to spawn kiro-cli: ... Is it installed?`
- Timeout: `kiro-cli ACP timed out after N seconds`
- Auth failure or no API key: return Kiro's sanitized ACP/stderr error; do
  not print environment values.
- Tool approval failure: surface the exact Kiro error and mention that the
  isolated no-tool agent should prevent this in default mode.

## Security Notes

The provider defaults to no Kiro-native tools. `--trust-all-tools` is not
used in the implementation. If a future setting allows trusted Kiro tools,
it must be explicit and documented as unsafe relative to Sashiko's own tool
sandbox.

Kiro authentication precedence gives an active browser login priority over
`KIRO_API_KEY`.

## Environment Variables

| Variable | Purpose |
|---|---|
| `KIRO_API_KEY` | Kiro API-key authentication for ACP use |

Kiro may also use an existing browser login depending on the local
environment. CI and production deployments should run in a clean service
environment or validate identity with `kiro-cli whoami`.

## Configuration Examples

Default isolated Kiro provider:

```toml
[ai]
provider = "kiro-cli"
model = "claude-sonnet-4"
max_input_tokens = 180000

[ai.kiro_cli]
binary = "kiro-cli"
context_window_size = 200000
```

With a custom Kiro agent:

```toml
[ai]
provider = "kiro-cli"
model = "claude-sonnet-4"

[ai.kiro_cli]
agent = "my-sashiko-agent"
```

Custom agents are an advanced escape hatch. The default generated agent
remains the safe path because it disables Kiro native tools and uses
Sashiko's tool-call protocol instead.
